# Конструктор роадмапа учебной программы

> **Статус:** концептуальный документ — описывает инструмент для преподавателя поверх DST-агрегации.
> Связан с архитектурой микросервисов: `ARCHITECTURE.md`.

---

## Концепция

Текущий процесс формирования учебного плана для Топ-ИТ: преподаватель берёт список
из ~10–12 компетенций КРМ и «раскидывает» их по дисциплинам вручную. Самый трудоёмкий
шаг — построение карты компетенций и «ромашки».

**Идея конструктора:** инвертировать процесс.

```
Сейчас:   Компетенции (КРМ) → преподаватель руками → Дисциплины
Предложение: Преподаватель строит дисциплины → система связывает с компетенциями
```

Преподаватель работает с тем, что он понимает — с конкретными технологиями и темами.
Система занимается формальным маппингом: какая тема закрывает какой индикатор,
какой индикатор относится к какой компетенции, есть ли пробелы в покрытии.

## Граф обучения: структура данных

### Узлы

Основная единица графа — **индикатор** из КРМ, не компетенция целиком.
КРМ уже даёт трёхуровневую структуру индикаторов:

```
Компетенция ПК-1: "Способен разрабатывать серверные приложения"
  ├── ИД-1.1 (базовый)     — пишет простые HTTP-обработчики на фреймворке
  ├── ИД-1.2 (средний)     — проектирует REST API с авторизацией и документацией
  └── ИД-1.3 (продвинутый) — разрабатывает высоконагруженные микросервисы
```

Каждый индикатор = узел графа с атрибутами:
```json
{
  "id": "ИД-1.2",
  "competency_id": "ПК-1",
  "level": "medium",
  "description": "проектирует REST API с авторизацией",
  "example_technologies": ["FastAPI", "Spring Boot", "JWT"],
  "dsт_betp": 0.82,          // актуальность из MIS (DST Level 2)
  "assigned_discipline": "Веб-разработка",
  "semester": 3
}
```

### Рёбра

Два типа рёбер:

| Тип | Смысл | Источник |
|-----|-------|---------|
| **prerequisite** | ИД-1.1 → ИД-1.2 (базовый должен быть до среднего) | Автоматически из уровней КРМ |
| **co-requisite** | ИД-1.2 ↔ ИД-3.1 (нужны параллельно) | Задаётся преподавателем или выводится из domain_markers |

**Автоматические prereq-рёбра внутри компетенции:**
```
базовый → средний → продвинутый
```

**Кросс-компетентностные prereq-рёбра** выводятся из технологического графа:
```
Технология "FastAPI" требует "Python (базовый)"
→ ИД-1.2 (FastAPI) → prereq → ИД-Python.1 (основы Python)
→ если ИД-Python.1 в семестре 3, а ИД-1.2 в семестре 2 → ошибка!
```

### Слои (семестры)

Граф ориентирован по семестрам. Корректный граф = направленный ациклический граф (DAG):

```
Семестр 1 ──▶ Семестр 2 ──▶ Семестр 3 ──▶ ... ──▶ Семестр 8
   │               │               │
[ИД-1.1]       [ИД-2.1]       [ИД-1.2]
[ИД-Py.1]      [ИД-Py.2]      [ИД-3.1]
```

## Интерфейс преподавателя: «конструктор»

### Рабочий процесс

```
1. Выбрать роли КРМ (из согласованной ЦКМ)
2. Создать дисциплины (название, семестр, количество часов)
3. Для каждой дисциплины добавить технологии и темы
4. Система: автоматически связывает технологии с индикаторами КРМ
5. Система: показывает что покрыто, что нет, какие конфликты
6. Преподаватель: корректирует, экспортирует карту компетенций и ромашку
```

### Шаг 3 подробнее: добавление технологий

Преподаватель открывает дисциплину «Веб-разработка» и добавляет:
```
Темы/технологии: FastAPI, REST API, JWT, Docker
```

Система немедленно отвечает по каждой:

```
┌─────────────┬───────────────────────────┬─────────┬──────────────────────────┐
│ Технология  │ Маппинг на индикатор      │ BetP    │ Подсказка                │
├─────────────┼───────────────────────────┼─────────┼──────────────────────────┤
│ FastAPI     │ ИД-1.2 (ПК-1, средний)   │ 0.82 ↑  │ Актуально                │
│ REST API    │ ИД-1.2 (ПК-1, средний)   │ 0.89 ↑  │ Актуально                │
│ JWT         │ ИД-4.1 (ПК-4, базовый)   │ 0.71    │ Покрывает безопасность   │
│ Docker      │ ИД-7.1 (ПРПК-2, базовый) │ 0.88 ↑  │ Актуально                │
│             │                           │         │ ⚠ Prereq: Linux basics   │
└─────────────┴───────────────────────────┴─────────┴──────────────────────────┘
```

### Подсказки системы (четыре типа)

#### 1. Актуальность навыка
DST Level 2 из MIS: BetP по вакансиям для данной роли.
```
🟢 FastAPI (BetP=0.82) — встречается в 82% вакансий "Backend Developer"
🟡 Flask   (BetP=0.44) — снижается, рассмотрите FastAPI вместо него
🔴 Django REST Framework (BetP=0.21, m(F)=0.51) — устаревает в данном контексте
```

#### 2. Маппинг на компетенции
Если технология не находит маппинга в КРМ ЦКМ:
```
⚠ "Redis" — не покрывается ни одной компетенцией в вашей ЦКМ.
  Ближайшие варианты: ПК-5 "Работа с БД" (BetP совместного появления: 0.67)
  Добавить в ЦКМ? [да / пропустить]
```

#### 3. Связность (prereq-конфликты)
```
❌ Конфликт: "FastAPI" в Семестре 2 (Дисциплина: Веб-разработка)
   требует: "Python базовый" (ИД-Python.1)
   но "Python базовый" → Семестр 3 (Дисциплина: Программирование)
   Решение: переместить "Программирование" на Семестр 1 или 2
```

#### 4. Пробелы в покрытии ЦКМ
```
⚠ Не покрыто: ИД-2.3 (ПК-2, продвинутый) — "оптимизирует алгоритмы под нагрузку"
   BetP = 0.79 — высокоприоритетный индикатор для роли "Инженер-программист"
   Рекомендуется добавить в: Семестр 5–6 (после покрытия ИД-2.1, ИД-2.2)
```

## Анализ пробелов: три типа

```
1. UNCOVERED — индикатор ЦКМ не назначен ни одной дисциплине
   Источник: статический обход графа (все узлы без assigned_discipline)

2. BROKEN PREREQUISITE — дисциплина A требует знаний из дисциплины B,
   но B идёт позже A в семестровом расписании
   Источник: топологическая сортировка (поиск циклов и нарушений порядка)

3. MARKET DEFICIT — индикатор покрыт в РПД, но BetP(технологий) значительно
   упал с момента последнего обновления, или m(F) > 0.5
   Источник: DST Level 2 из MIS при ежегодном пересчёте
```

## Выходные артефакты (генерируются автоматически)

### Карта компетенций
Матрица «дисциплина × индикатор», генерируется из рёбер графа:

```
                  │ ИД-1.1 │ ИД-1.2 │ ИД-1.3 │ ИД-2.1 │ ИД-4.1 │
──────────────────┼────────┼────────┼────────┼────────┼────────┤
Программирование  │   ✓    │        │        │   ✓    │        │
Веб-разработка    │        │   ✓    │        │        │   ✓    │
Архитектура ПО    │        │        │   ✓    │        │        │
```

### Ромашка
Радиальная визуализация: центр = роль КРМ, лепестки = компетенции,
каждый лепесток разделён на сегменты по индикаторам.
Цвет сегмента: зелёный (покрыт) / жёлтый (частично) / красный (не покрыт).

### Отчёт pre-validation
Документ в формате проверки Топ-ИТ до отправки оператору:
```
✓ Доля ПК топ-уровня: 71% (≥50% ✓, ≥70% ✓)
✓ Индикаторы базового уровня: 88% покрыты (≥80% ✓)
✗ Индикаторы среднего уровня: 54% покрыты (≥60% ✗)
  → Не покрыто: ИД-1.2, ИД-3.2, ИД-5.1
✓ Конфликтов prereq: 0
⚠ Устаревших технологий: 2 (Flask в ПК-1, Keras в ПК-3) — рекомендуется пересмотреть
```

## Связь с DST

| Подсказка конструктора | DST-механизм | Уровень |
|------------------------|--------------|---------|
| Актуальность технологии | BetP из VAC/EXP агрегации | Level 2 |
| Технология устарела | m(F) > 0.5 из DST | Level 2 |
| Рыночный приоритет индикатора | Δ = BetP − supply | Level 1 |
| Конфликт партнёр/рынок | K ≥ τK при комбинировании VAC+EXP | Level 1 |
| Неопределённость оценки | m(U) > τΘ | Level 1–2 |

## Что НЕ делает DST в конструкторе

DST отвечает на вопрос **«что»** (какие технологии актуальны, какие компетенции в дефиците).
DST **не отвечает** на вопрос **«как»** (в каком порядке учить, в какую дисциплину положить).

Порядок и связность — это задача **топологической сортировки графа индикаторов**.
Распределение по дисциплинам — это задача **преподавателя** с подсказками системы.

```
DST → «Усилить FastAPI, убрать Flask»
Граф → «FastAPI можно только после Python-базовый»
Конструктор → «Положи FastAPI в Семестр 3, если Python в Семестре 1»
```
