## Часть 1: Трёхуровневая архитектура

Система принятия решений работает на трёх иерархических уровнях. Каждый уровень запускается последовательно: результаты верхнего уровня управляют работой нижнего.

```
╔══════════════════════════════════════════════════════════════╗
║  УРОВЕНЬ 0 — Профессии / Специальности                       ║
║                                                              ║
║  Вопрос: Какие кластеры компетенций критичны для             ║
║  конкретной профессии (BackEnd, Data Scientist, DevOps…)?    ║
║                                                              ║
║  Входные данные:                                             ║
║  • Вакансии, профилированные по специальностям               ║
║  • Компетентностные модели из ФГОС/профстандартов            ║
║                                                              ║
║  Выход: Ранжированный список кластеров с весами              ║
║         критичности для каждой специальности                  ║
╚════════════════════════╦═════════════════════════════════════╝
                         ↓ для кластеров с высоким весом
╔════════════════════════╩═════════════════════════════════════╗
║  УРОВЕНЬ 1 — Кластеры компетенций (Θ)                        ║
║                                                              ║
║  Вопрос: Достаточно ли кластер представлен в РПД             ║
║  относительно требований рынка?                              ║
║                                                              ║
║  Источники: VAC + EXP + FC + STD                             ║
║  Выход: m*(T), m*(U), m*(F), BetP, Δ, рекомендация          ║
╚════════════════════════╦═════════════════════════════════════╝
                         ↓ только для кластеров с решением УСИЛИТЬ
╔════════════════════════╩═════════════════════════════════════╗
║  УРОВЕНЬ 2 — Навыки и технологии (u ∈ Θ)                     ║
║                                                              ║
║  Вопрос: Какие конкретные навыки внутри кластера             ║
║  покрыты плохо и требуют включения в дисциплины?             ║
║                                                              ║
║  Источники: те же, но по конкретному навыку                  ║
║  Выход: приоритизированный список навыков для включения      ║

```

### Уровень 0 — Профессии

**Цель:** До запуска основного алгоритма понять, *для кого* мы обновляем программу.

**Пример профилирования:**

| Кластер | BackEnd Dev | Data Scientist | DevOps | 1С-разработчик |
|---------|:-----------:|:--------------:|:------:|:--------------:|
| Python | 0.7 | 0.95 | 0.4 | 0.05 |
| CI/CD | 0.8 | 0.3 | 0.98 | 0.1 |
| Java | 0.85 | 0.2 | 0.3 | 0.05 |
| Реляц. БД | 0.9 | 0.6 | 0.5 | 0.7 |
| 1С | 0.05 | 0.0 | 0.0 | 0.98 |

Вес кластера для программы = взвешенная сумма по всем профессиям, которым обучает вуз.

**Как получить профили профессий:**
- NLP-кластеризация вакансий по названиям должностей
- Анализ связки «должность → требуемые навыки» через совместную встречаемость
- Экспертная валидация профилей

### Уровень 1 — Кластеры

Основной уровень. Подробно описан в `ALGORITHM_CURRENT.md` (шаги 0–6).  
В идеальной версии добавляются: четвёртый источник STD, правило Ягера, автокалибровка параметров.

### Уровень 2 — Навыки

Запускается **только** для кластеров, получивших решение **УСИЛИТЬ** на Уровне 1.

**Фрейм:** `Θ_u = {T, F}` — бинарный (навык нужно включить или нет).

**Алгоритм:** аналогичен Уровню 1, но:
- Область рассмотрения = навыки внутри кластера, а не все кластеры
- `supply_u` = часы на конкретный навык в данном кластере
- `N` в формуле BetP = число навыков в кластере (обычно 5–15)
- `averageScore` считается по частоте совместного упоминания навыка с кластером в вакансиях

**Выход:** список навыков с рекомендациями «включить», «усилить», «сохранить».

## Часть 2: Улучшения для каждого шага

### Шаг 1 (κ): Автоматический подбор λ

**Текущее состояние:** `λ = 15 / 5 / 2` зашито вручную.

**Идеальное состояние:** λ вычисляется из реальных данных по медиане объёмов.

```
Метод 1 (простой): λ = median(n_eff_по_всем_кластерам) / 10

  При n_eff = median → κ = 1/(1 + 1/10) ≈ 0.91  ← высокая уверенность
  При n_eff = median/2 → κ ≈ 0.83
  При n_eff = 0 → κ = 0

Метод 2 (робастный): λ = percentile_25(n_eff) / 10
  Защита от завышения λ из-за популярных технологий (Python, Java)

Метод 3 (точный): подбор λ*, минимизирующего ошибку ретроспективного предсказания
  Берём данные года T, предсказываем год T+3, сравниваем с реальностью
  λ* = argmin_λ MSE(предсказание, реальность)
```

**Периодичность пересчёта:** раз в год или при изменении объёма данных более чем на 30%.

### Шаг 2 (BPA): averageScore из текста вакансий

**Текущее состояние:** `averageScore` — фиксированное предположение.

**Идеальное состояние:** вычисляется через NLP-анализ текста вакансий.

```
averageScore(кластер, вакансия) = f(позиция, частота, контекст, зарплата)

Факторы важности навыка в вакансии:
  • Позиция в тексте:    заголовок → 1.0; «обязательно» → 0.9; «желательно» → 0.5
  • Частота:             упоминается N раз → score × log(N+1)
  • Зарплатный сигнал:   вакансии с навыком платят на X% больше → весовой коэффициент
  • Контекст:            «опыт 3+ лет» → 1.0; «знакомство» → 0.4

averageScore(кластер) = mean(score по всем вакансиям, где кластер упомянут)
```

**Для RPD (STD-источник):**
```
averageScore_std(кластер) = часы_по_кластеру / общий_объём_программы
                          = supply  ← то, что сейчас есть
```

Но RPD может давать и `m(F)` > 0, если дисциплина явно «устарела» по меткам или исключена из нового ФГОС.

### Шаг 3 (дисконтирование): Калибровка весов w

**Текущее состояние:** `w = 0.8 / 0.9 / 0.6` задаются экспертно.

**Идеальное состояние:** веса калибруются на исторических данных.

```
Процедура калибровки:

1. Берём данные за годы [T-5 … T-1]
2. Обучаем алгоритм с разными комбинациями w_vac, w_exp, w_fc
3. Оцениваем качество предсказания по метрике:
      accuracy = доля кластеров, для которых рекомендация совпала
                 с фактическим изменением программы через 3 года
4. w* = argmax accuracy

Ожидаемый диапазон:
  w_vac ∈ [0.70, 0.90]  — зависит от качества парсинга вакансий
  w_exp ∈ [0.80, 0.95]  — зависит от репрезентативности выборки экспертов
  w_fc  ∈ [0.50, 0.70]  — зависит от горизонта и методологии прогнозов
```

**Дополнение:** веса могут зависеть от типа кластера:
- Зрелые технологии (Java, SQL): прогнозы точнее → `w_fc` выше
- Новые технологии (LLM, Rust): прогнозы ненадёжнее → `w_fc` ниже

### Шаг 4 (комбинирование): Правило Ягера при высоком конфликте

> Это ключевое отличие идеального алгоритма от текущей реализации.

#### Когда применяется

При K ≥ τK = 0.4 правило Демпстера ведёт себя нестабильно: нормировка на `(1−K)` может дать контринтуитивный результат — уверенность в одном из исходов резко возрастает именно потому, что конфликт велик.

**Пример:**
```
Источник 1: m(T) = 0.9,  m(U) = 0.1,  m(F) = 0
Источник 2: m(T) = 0,    m(U) = 0.1,  m(F) = 0.9

K = 0.9 × 0.9 + 0 × 0 = 0.81  ← огромный конфликт!
norm = 1 / (1 − 0.81) = 5.26

Демпстер: m_out(T) = (0.9×0 + 0.9×0.1 + 0.1×0) / 0.19 = 0.09/0.19 = 0.47
          m_out(F) = (0×0.9 + 0×0.1   + 0.1×0.9) / 0.19 = 0.09/0.19 = 0.47

Результат: оба исхода равновероятны — алгоритм "застрял".
```

#### Формула правила Ягера

Ягер не нормирует: конфликтная масса K уходит в полное незнание Θ.

```
Шаг 1: Вычислить "сырые" массы как у Демпстера (без нормировки):
  raw(T) = Σ_{B∩C=T} m₁(B) × m₂(C)  =  m₁(T)×m₂(T) + m₁(T)×m₂(U) + m₁(U)×m₂(T)
  raw(U) = Σ_{B∩C=U} m₁(B) × m₂(C)  =  m₁(U)×m₂(U)
  raw(F) = Σ_{B∩C=F} m₁(B) × m₂(C)  =  m₁(F)×m₂(F) + m₁(F)×m₂(U) + m₁(U)×m₂(F)
  raw(∅) = K = m₁(T)×m₂(F) + m₁(F)×m₂(T)    ← конфликтная масса

Шаг 2: Конфликтная масса переводится в неопределённость Θ:
  m_Y(T) = raw(T)
  m_Y(U) = raw(U) + K           ← К уходит не в нормировку, а в "не знаю"
  m_Y(F) = raw(F)

  Сумма: m_Y(T) + m_Y(U) + m_Y(F) = raw(T) + raw(U) + K + raw(F) = 1 ✓
```

#### Сравнение Демпстер vs Ягер на примере

```
Исходные данные:
  Вакансии: m(T)=0.7, m(U)=0.2, m(F)=0.1
  Эксперты: m(T)=0.1, m(U)=0.2, m(F)=0.7

K = 0.7×0.7 + 0.1×0.1 = 0.49 + 0.01 = 0.50  ← высокий конфликт!

──────────────────────────────────────────────────────
Правило Демпстера (нормировка на 1−K = 0.5):

  raw(T) = 0.7×0.1 + 0.7×0.2 + 0.2×0.1 = 0.07+0.14+0.02 = 0.23
  raw(F) = 0.1×0.7 + 0.1×0.2 + 0.2×0.7 = 0.07+0.02+0.14 = 0.23
  raw(U) = 0.2×0.2 = 0.04

  m_D(T) = 0.23 / 0.5 = 0.46
  m_D(U) = 0.04 / 0.5 = 0.08
  m_D(F) = 0.23 / 0.5 = 0.46

  → «Источники поровну: включить или исключить?» Ложная уверенность!

──────────────────────────────────────────────────────
Правило Ягера (конфликт → неопределённость):

  m_Y(T) = 0.23
  m_Y(U) = 0.04 + 0.50 = 0.54   ← конфликт честно стал незнанием
  m_Y(F) = 0.23

  → BetP = 0.23 + 0.54/N  ←  низкое, с большой неопределённостью
  → Результат: ВАРИАТИВНОСТЬ (данные противоречивы, нужна экспертиза)
  → Это честнее: алгоритм признаёт, что не знает ответа!
```

#### Сравнительная таблица правил комбинирования

| Правило | Формула конфликта | Когда применять | Особенность |
|---------|-------------------|-----------------|-------------|
| **Демпстер** | K нормируется: ÷(1−K) | K < 0.4 | Быстро усиливает уверенность при согласии |
| **Ягер** | K → m(Θ) | K ≥ 0.4 | Честно признаёт неопределённость, не «насилует» данные |
| **Дюбуа-Прада** | K частично → m(A∪B) | K умеренный | Компромисс: часть конфликта остаётся в конкретных гипотезах |
| **TBM** | Открытый мир: ∅ допустима | Исследование | Позволяет полное незнание как допустимое состояние |

**Рекомендация для системы:** применять адаптивно:
```
if K < 0.2:  правило Демпстера (чистый случай)
if K < 0.4:  правило Демпстера с предупреждением
if K ≥ 0.4:  правило Ягера + пометка "требует экспертизы"
if K ≥ 0.7:  правило Ягера + принудительная ВАРИАТИВНОСТЬ
```

### Шаг 4 (комбинирование): Четвёртый источник — STD (РПД)

В текущей реализации РПД — это только `supply` (скаляр).  
В идеале РПД является полноценным независимым источником DST.

**Как строится BPA для STD:**

```
Пусть hours_cluster = часы по кластеру в РПД
     hours_total    = общий объём программы

Доля кластера:  r_std = hours_cluster / hours_total

Если r_std >> BetP_других_источников:
  → Программа переоценивает этот кластер
  → m_std(F) = κ_std × (r_std − типичная_доля)   ← сигнал избытка

Если r_std << BetP_других_источников:
  → Программа недооценивает этот кластер
  → m_std(T) = κ_std × BetP_других             ← сигнал дефицита

m_std(U) = 1 − m_std(T) − m_std(F)
```

**Порядок комбинирования с четырьмя источниками:**
```
((VAC ⊕ EXP) ⊕ FC) ⊕ STD
```

STD добавляется последним, поскольку является «консервативным» источником — программа меняется медленно.

### Шаг 5 (BetP): Динамический N_CLUSTERS

**Текущее состояние:** `N = 25` хардкод.

**Идеальное состояние:**
```
N = COUNT(DISTINCT cluster_id) FROM academic_program WHERE program_id = current
```

Или на Уровне 2:
```
N = COUNT(skills) WHERE cluster_id = current_cluster
```

**Дополнение: двойная оценка:**

Вместо единственного BetP, предоставлять лицу, принимающему решение, интервал:
```
Нижняя граница:  Bel(T)  = m*(T)            (минимальная уверенность)
Точечная оценка: BetP(T) = m*(T) + m*(U)/N
Верхняя граница: Pl(T)   = 1 − m*(F)        (максимальная уверенность)

Интерпретация: "Важность кластера оценивается в диапазоне [Bel; Pl],
               наиболее вероятная оценка BetP"
```

### Шаг 6 (решение): Адаптивные пороги

**Текущее состояние:** `τΔ = 0.15`, `τK = 0.4`, `τΘ = 0.15` — экспертные.

**Идеальное состояние:** пороги откалиброваны на ретроспективных данных.

**Процедура калибровки τΔ:**
```
1. Берём N исторических случаев, где были сделаны изменения программы
2. Для каждого случая вычисляем Δ_ретро на момент решения
3. Строим ROC-кривую: TP/FP при разных значениях τΔ
4. τΔ* = порог, максимизирующий F1-score (или другую целевую метрику)
```

**Зависимость τΔ от контекста:**

| Ситуация | Рекомендуемый τΔ | Обоснование |
|----------|-----------------|-------------|
| Программа обновляется раз в 5 лет | 0.10 | Реагировать на меньший дефицит |
| Программа обновляется ежегодно | 0.20 | Выше порог — только значимые изменения |
| Новая программа | 0.05 | Максимальная чувствительность |
| Стабильная программа | 0.25 | Консервативная стратегия |

## Часть 3: Схема данных для идеального алгоритма

### Требуемые таблицы

```sql
-- Уровень 0: Профессии и их связь с кластерами
CREATE TABLE professions (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100),
    description TEXT
);

CREATE TABLE profession_cluster_weights (
    profession_id INT REFERENCES professions(id),
    cluster_id    INT REFERENCES skills_group(id),
    weight        DECIMAL(3,2),  -- важность кластера для профессии
    source        VARCHAR(30)    -- 'fgos', 'expert', 'vacancy_analysis'
);

-- Уровень 1: Уже есть в AcademicSupport (частично)
-- skills_group → кластеры
-- work_skill   → навыки
-- employer     → работодатели-эксперты
-- vacancy      → вакансии
-- rpd (нет)   → нужна таблица для покрытия РПД

CREATE TABLE rpd_coverage (
    program_id    INT,
    cluster_id    INT REFERENCES skills_group(id),
    skill_id      INT REFERENCES work_skill(id),
    hours         INT,
    semester      INT,
    academic_year INT,
    updated_at    TIMESTAMP
);

-- Прогнозы: нужна доработка существующей таблицы
-- Добавить поля direction (positive/negative/mixed) и confidence
ALTER TABLE forecast ADD COLUMN direction VARCHAR(10);  -- 'positive', 'negative'
ALTER TABLE forecast ADD COLUMN confidence DECIMAL(3,2);

-- Результаты DST: сохранять историю рекомендаций
CREATE TABLE dst_results (
    id            SERIAL PRIMARY KEY,
    cluster_id    INT REFERENCES skills_group(id),
    run_date      DATE,
    m_t           DECIMAL(5,4),
    m_u           DECIMAL(5,4),
    m_f           DECIMAL(5,4),
    k_conflict    DECIMAL(5,4),
    betp          DECIMAL(5,4),
    supply        DECIMAL(5,4),
    delta         DECIMAL(5,4),
    recommendation VARCHAR(20),
    params_json   JSONB  -- λ, w, τ на момент расчёта
);
```

| averageScore из TF-IDF | Более точная BPA для вакансий; различие между «обязательно» и «желательно» |
| Калибровка w | Повышение точности предсказаний на 10–20% (исходя из аналогичных работ) |
| Калибровка λ | Устранение систематической ошибки при малом/большом объёме данных |
| STD как источник | Алгоритм «видит» текущую программу, а не получает её как внешний параметр |
| Уровень 2 | Конкретные навыки для включения, не только кластеры |
| Уровень 0 | Рекомендации, учитывающие профиль специальности |
| Адаптивные τ | Меньше ложных срабатываний при обновлении программ |
| Контекстные кластеры | Устранение смешения разных доменов применения одной технологии |
| Обнаружение сдвига | Выявление перераспределения внутри стабильного кластера |
| Онтология технологий | Учёт косвенных упоминаний (Flask → Python) |
| Учёт версий | Автоматическое снижение effective_supply при отставании РПД по версиям |

## Часть 6: Дополнительные архитектурные проблемы и их решения

### 6.1 Контекстно-зависимые кластеры

**Проблема:** Одна технология используется принципиально по-разному в разных профессиях. «Python для бэкенда» и «Python для аналитики» — это разные дисциплины, разные навыки, разные задачи в РПД. Если объединить их в один кластер «Python», данные из двух доменов будут смешиваться и искажать BPA.

**Решение: кластер = технология + домен применения**

```
Вместо:   Python (один кластер)

Ввести:   Python/Backend       — Flask, FastAPI, Django, REST API
          Python/DataScience   — pandas, NumPy, Jupyter, sklearn, Colab
          Python/Automation    — pytest, Selenium, скрипты, CI/CD
```

**Как определить домен при парсинге вакансии:**

Смотрим на **со-встречаемые навыки** в той же вакансии. Кластер вакансии определяется доминирующим доменом:

```
Python + (Flask, FastAPI, REST)     → Python/Backend
Python + (pandas, sklearn, Jupyter) → Python/DataScience
Python + (pytest, Jenkins, Docker)  → Python/Automation
```

**Реализация в БД:**

```sql
CREATE TABLE skills_group (
    id          SERIAL PRIMARY KEY,
    technology  VARCHAR(100),   -- 'Python'
    domain      VARCHAR(100),   -- 'Backend', 'DataScience', 'Automation'
    name        VARCHAR(200)    -- 'Python/Backend'
);

-- Маркеры доменов: сигнальные навыки, указывающие на домен
CREATE TABLE domain_markers (
    domain      VARCHAR(100),
    marker_skill VARCHAR(100),  -- 'Flask', 'pandas'...
    weight      DECIMAL(3,2)    -- сила сигнала
);
```

**Связь с Уровнем 0 (профессии):**

Профиль профессии теперь задаётся по (technology, domain):
- BackEnd Dev → Python/Backend: 0.85
- Data Scientist → Python/DataScience: 0.95
- BackEnd Dev → Python/DataScience: 0.20 ← небольшой вес, но не ноль

**Ограничение:** Размытые вакансии без чёткого домена — нужно либо относить к нескольким доменам с дробным весом, либо иметь кластер «Python/General» как fallback.

### 6.2 Обнаружение внутреннего перераспределения

**Проблема:** Кластер «JS-фреймворки» в целом стабилен: React −20%, Vue +20%. Уровень 1 видит Δ ≈ 0 и выдаёт СТАБИЛИЗАЦИЯ. Уровень 2 не запускается. Реальное изменение на рынке остаётся незамеченным.

**Решение: второй триггер Уровня 2 — «обнаружение сдвига»**

Уровень 2 запускается при **двух условиях** (не только при УСИЛИТЬ):

```
Триггер 1 (дефицит):      Δ_кластера > τΔ   → УСИЛИТЬ
                           → Уровень 2: найти, какие навыки покрыты плохо

Триггер 2 (перераспределение):
    |Δ_кластера| ≤ τΔ     (кластер в целом стабилен)
    AND Var(Δ_навыков) > τ_var  (разброс спроса по суб-навыкам высок)
    → Уровень 2: обнаружить, какие навыки растут/падают
```

**Как вычислить Var(Δ_навыков):**

```
Для каждого навыка u ∈ кластер C:
    BetP_u = агрегированная важность навыка u
    Δ_u    = BetP_u − supply_u

Var(Δ) = среднеквадратичное отклонение {Δ_u} по всем u в C
τ_var  = порог (подбирается эмпирически, например 0.15)
```

**Вывод Уровня 2 при триггере «перераспределение»:**

| Навык | BetP | supply | Δ | Тренд | Рекомендация |
|-------|------|--------|---|-------|--------------|
| React | 0.38 | 0.50 | −0.12 | ↓ | Сократить |
| Vue | 0.52 | 0.30 | +0.22 | ↑ | Усилить |
| Angular | 0.10 | 0.20 | −0.10 | ↓ | Сократить |

**Ключевое условие:** Прогнозы (FC-источник) должны быть на уровне отдельных навыков (React, Vue), а не только на уровне кластера. Это требование к данным, а не к алгоритму.

### 6.3 Онтология технологий: учёт косвенных упоминаний

**Проблема:** Вакансия пишет «нужен Flask-разработчик» — слова «Python» нет. При прямом поиске вакансия не попадёт в кластер Python. Реальный спрос на Python занижается.

**Решение: граф технологических зависимостей**

Строится направленный граф «A → B» (читать: «использование A подразумевает владение B»):

```
Flask       → Python      (вес 1.0)
FastAPI     → Python      (вес 1.0)
Django      → Python      (вес 1.0)
pandas      → Python      (вес 0.9)
NumPy       → Python      (вес 0.9)

Spring Boot → Java        (вес 1.0)
Hibernate   → Java        (вес 0.9)

React       → JavaScript  (вес 1.0)
Vue         → JavaScript  (вес 1.0)
Next.js     → JavaScript  (вес 0.95)
Next.js     → React       (вес 0.9)    ← транзитивная зависимость

.NET        → C#          (вес 0.95)
```

**Формула эффективного подсчёта упоминаний:**

```
count_eff(Python) =   1.0 × count_direct("Python")
                    + 1.0 × count("Flask")
                    + 1.0 × count("FastAPI")
                    + 1.0 × count("Django")
                    + 0.9 × count("pandas")
                    + 0.9 × count("NumPy")
                    + ...  (все узлы, ведущие в Python)
```

**Реализация в БД:**

Каждый навык хранится в справочной таблице с числовым ID. `technology_ontology` ссылается на записи через FK — это обеспечивает целостность, позволяет одному «родителю» иметь множество «детей» (один-ко-многим), и исключает дублирование строковых имён.

```sql
-- Справочник всех навыков/технологий (уже частично есть как work_skill)
CREATE TABLE work_skill (
    id          SERIAL PRIMARY KEY,
    name        VARCHAR(100) NOT NULL UNIQUE,  -- 'Flask', 'Python', 'React'
    skill_type  VARCHAR(30)                    -- 'framework', 'language', 'tool'
);

-- Граф зависимостей: child требует/подразумевает parent
-- Один parent (Python) ← много child'ов (Flask, FastAPI, Django, pandas…)
CREATE TABLE technology_ontology (
    id            SERIAL PRIMARY KEY,
    child_id      INT REFERENCES work_skill(id) ON DELETE CASCADE,  -- Flask
    parent_id     INT REFERENCES work_skill(id) ON DELETE CASCADE,  -- Python
    weight        DECIMAL(3,2) NOT NULL DEFAULT 1.0,
    relation      VARCHAR(30)  NOT NULL,  -- 'requires', 'implies', 'part_of'
    UNIQUE (child_id, parent_id)
);
```

**Почему каждая строка = одно ребро графа:**

Каждая запись в `technology_ontology` — это одна связь «child требует parent». Один родитель (JavaScript) с тремя дочерними фреймворками хранится как три отдельные строки:

```
 id | child_id | parent_id | weight | relation
----|----------|-----------|--------|----------
  1 |    5     |     4     |  1.0   | requires   ← React   → JavaScript
  2 |    6     |     4     |  1.0   | requires   ← Vue     → JavaScript
  3 |    7     |     4     |  1.0   | requires   ← Angular → JavaScript
```

В реляционных БД массив нельзя хранить в одном поле — отношение «один-ко-многим» всегда раскладывается в несколько строк.

- Вопрос «что подразумевает JS?» → `WHERE parent_id = 4` → React, Vue, Angular
- Вопрос «что нужно для React?» → `WHERE child_id = 5` → JavaScript

**Зачем нужны оба поля одновременно — потому что это граф, а не дерево.** У одного `child` может быть несколько `parent`'ов:

```
 id | child_id | parent_id | relation
----|----------|-----------|----------
  4 |    8     |     4     | requires   ← Next.js → JavaScript
  5 |    8     |     5     | requires   ← Next.js → React      (тоже нужен!)
```

Next.js требует и JavaScript, и React — два ребра, два `parent_id` в двух строках. Без явного `parent_id` в каждой строке это невозможно выразить.

Пример данных:
```sql
-- work_skill: id=1 Python, id=2 Flask, id=3 FastAPI, id=4 JavaScript, id=5 React, id=6 Vue, id=7 Angular, id=8 Next.js
INSERT INTO technology_ontology (child_id, parent_id, weight, relation) VALUES
    (2, 1, 1.0, 'requires'),   -- Flask     → Python
    (3, 1, 1.0, 'requires'),   -- FastAPI   → Python
    (5, 4, 1.0, 'requires'),   -- React     → JavaScript
    (6, 4, 1.0, 'requires'),   -- Vue       → JavaScript
    (7, 4, 1.0, 'requires'),   -- Angular   → JavaScript
    (8, 4, 1.0, 'requires'),   -- Next.js   → JavaScript
    (8, 5, 0.9, 'requires');   -- Next.js   → React (тоже нужен)
```

Запрос для `count_eff(Python, id=1)` через граф:
```sql
SELECT
    SUM(v.count * o.weight) AS count_eff
FROM vacancy_skill v
JOIN technology_ontology o ON v.skill_id = o.child_id
WHERE o.parent_id = 1  -- Python
UNION ALL
SELECT COUNT(*) FROM vacancy_skill WHERE skill_id = 1;  -- прямые упоминания
```

**Как поддерживать граф:**
- Начальное заполнение — вручную экспертами
- Автоматическое дополнение — через статистику совместной встречаемости: если `child_id` встречается в вакансиях совместно с `parent_id` в 90%+ случаев → ребро подтверждается или создаётся
- Пересмотр раз в год или при появлении новых фреймворков

**Ограничение:** Транзитивные зависимости нужно применять с осторожностью. Если Next.js → React → JavaScript, то при расчёте JavaScript не нужно считать Next.js дважды — JOIN должен ограничиваться одним уровнем глубины, либо использоваться рекурсивный CTE с дедупликацией вакансий.

### 6.4 Учёт версий и актуальности технологий

**Проблема:** РПД покрывает .NET 6, вакансии требуют .NET 10. Формально кластер «.NET» покрыт (supply > 0), фактически — выпускники не готовы к работе с актуальной версией.

**Решение: коэффициент актуальности версии, корректирующий effective_supply**

**Шаг 1: извлечение версий**

Регулярные выражения при парсинге вакансий и РПД:
```
\.NET\s*(\d+)         → .NET 6, .NET 8, .NET 10
Python\s*3\.(\d+)     → Python 3.9, Python 3.12
Java\s*(\d+)          → Java 11, Java 17, Java 21
Spring Boot\s*([\d.]+) → Spring Boot 3.2
```

**Шаг 2: вычисление двух рыночных ориентиров**

Одной «версии рынка» недостаточно: самая свежая версия (.NET 10) — это передовой край, но большинство production-систем работают на более ранней. Поэтому собираем две метрики:

```
max_version    = максимальная версия, встречающаяся в вакансиях
                 («что хотят самые передовые работодатели»)

median_version = медианная версия по всем вакансиям, где указана версия
                 («что реально используется большинством компаний»)
```

Пример для .NET на текущем рынке:
```
Вакансии с указанием версии: [6, 6, 7, 8, 8, 8, 9, 10, 10]
max_version    = 10
median_version = 8  ← большинство всё ещё на 8
```

**Шаг 3: вычисление version_gap и version_currency**

Допустимость отставания определяется **относительно медианной версии**, а не максимальной. Если РПД учит .NET 8, а рынок медианно требует .NET 8 — это нормально, даже если существует .NET 10.

```
version_gap = median_version(рынок) − version(РПД)
              (берём со знаком: отрицательный gap = РПД опережает медиану)

version_currency = f(version_gap):
    gap ≤ 0   → 1.00  (РПД актуально или опережает медиану)
    gap = 1   → 0.80  (небольшое отставание, допустимо)
    gap = 2   → 0.50  (предупреждение)
    gap = 3   → 0.25  (значительное отставание)
    gap ≥ 4   → 0.10  (критическое устаревание)
```

Дополнительно: если `version(РПД) < median_version` **и** `version(РПД) < max_version − 1` — это «двойной сигнал» устаревания, который можно отдельно пометить.

**Пример:**

| РПД | median_version | max_version | version_gap | version_currency | Статус |
|-----|----------------|-------------|-------------|------------------|--------|
| .NET 8 | .NET 8 | .NET 10 | 0 | 1.00 | ✅ Актуально |
| .NET 6 | .NET 8 | .NET 10 | 2 | 0.50 | ⚠️ Отстаёт от рынка |
| .NET 6 | .NET 10 | .NET 10 | 4 | 0.10 | 🔴 Устарело |
| Python 3.9 | Python 3.11 | Python 3.13 | 2 | 0.50 | ⚠️ Отстаёт от рынка |
| Java 17 | Java 17 | Java 21 | 0 | 1.00 | ✅ LTS актуально |

**Шаг 4: корректировка effective_supply**

```
effective_supply = supply × version_currency

Пример: supply = 0.30, version_gap = 2
effective_supply = 0.30 × 0.50 = 0.15

Δ = BetP − effective_supply → дефицит выявляется даже при формальном покрытии
```

**Реализация в БД:**

```sql
-- Версии в РПД
ALTER TABLE rpd_coverage ADD COLUMN skill_version VARCHAR(20);

-- Рыночные ориентиры по версиям (пересчитываются при обновлении данных)
CREATE TABLE skill_version_stats (
    skill_id         INT REFERENCES work_skill(id),
    snapshot_date    DATE,
    max_version      VARCHAR(20),     -- самая свежая на рынке
    median_version   VARCHAR(20),     -- медианная по вакансиям
    p25_version      VARCHAR(20),     -- 25-й процентиль (консервативный рынок)
    sample_size      INT,             -- число вакансий с указанной версией
    PRIMARY KEY (skill_id, snapshot_date)
);
```

**Ограничение:** Версии не всегда указываются явно ни в вакансиях, ни в РПД. Если версия не извлечена — `version_currency = 1.0` по умолчанию (не штрафуем за отсутствие информации). Для технологий без версионирования (например, «SQL», «Git») параметр не применяется. `median_version` требует достаточного объёма вакансий с явным указанием версии — при малом `sample_size` лучше использовать `max_version` со смягчёнными коэффициентами.
