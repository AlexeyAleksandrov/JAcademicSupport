# Текущая реализация DST-алгоритма: пошаговое описание

> **Статус:** реализовано в `dst-demo` (Java, без БД).  
> Связанные файлы: `SourceData.java`, `DSTAggregationService.java`, `DemoApplication.java`.  
> Старые пояснительные файлы (README.md, EXAMPLES_GUIDE.md) сохранены — они объясняют примеры, этот файл объясняет **сам алгоритм**.

---

## Обозначения

| Символ | Что означает |
|--------|-------------|
| `m(T)` | Масса «кластер важен» (True) |
| `m(U)` | Масса «не знаю» (Uncertainty / Θ) |
| `m(F)` | Масса «кластер не важен» (False) |
| `κ` | Коэффициент уверенности по объёму данных |
| `n_eff` | Эффективный объём источника |
| `λ` | Константа сглаживания (полудоверие) |
| `w` | Вес надёжности источника |
| `K` | Коэффициент конфликта между источниками |
| `Δ` | Показатель дефицита (BetP − supply) |
| `τΔ, τK, τΘ` | Пороги принятия решения |

## Шаг 0 — Входные данные

### 1. Название
Формирование объекта кластера из входных параметров.

### 2. Входные данные
Задаются вручную в `DemoApplication.java` через конструкторы `SourceData` и `Cluster`:

| Поле | Тип | Смысл |
|------|-----|-------|
| `totalCount` | int | Всего записей данного типа (все вакансии, все эксперты…) |
| `relevantCount` | int | Записей, относящихся к данному кластеру |
| `averageScore` | double [0;1] | Средняя «важность» кластера по данному источнику |
| `agreementLevel` | double [0;1] | Согласованность внутри источника (1.0 = все единодушны) |
| `weight` | double [0;1] | Вес надёжности источника `w` |
| `lambda` | double | Константа сглаживания `λ` |
| `currentRpdCoverage` | double [0;1] | Текущая доля кластера в учебной программе (`supply`) |
| `totalRpdHours` | int | Число часов, выделенных кластеру в РПД |

Для прогнозов со смешанным направлением используется фабричный метод:
```java
SourceData.mixedForecasts(total, growthCount, avgGrowthScore,
                          declineCount, avgDeclineScore, w, lambda)
```

### 3. Зачем этот шаг
Унифицировать разнородные источники (вакансии, эксперты, прогнозы) в единый формат перед вычислениями.

### 4. Формулы
Формул нет — только сбор данных в объект.

### 5. Что получаем
Объект `Cluster`, содержащий три объекта `SourceData` (вакансии, эксперты, прогнозы) и значение `supply`.

### 6. Что это даёт
Единая точка входа для алгоритма. Все последующие шаги читают только из этого объекта.

### 7. Ограничения и хардкод
- **Все значения задаются вручную.** Нет подключения к БД.
- `averageScore` для вакансий — **допущение** (0.75–0.85), не вычисляется из текста вакансий.
- `agreementLevel` для вакансий и прогнозов **всегда 1.0** — по упрощённой модели.
- Отсутствует источник `STD` (РПД/учебные планы) как независимый DST-источник; РПД учитывается только как `supply`.

### 8. Планы на будущее
- Загружать данные из PostgreSQL (`AcademicSupport`).
- Вычислять `averageScore` для вакансий через TF-IDF или веса упоминания навыков в тексте.
- Вычислять `agreementLevel` для прогнозов из разброса оценок аналитиков.
- Добавить РПД как полноценный DST-источник (`m_std(T)`, `m_std(U)`, `m_std(F)`).

## Шаг 1 — Вычисление κ (коэффициент уверенности)

### 1. Название
Оценка надёжности источника по объёму данных.

### 2. Входные данные
- `relevantCount` — количество записей по кластеру
- `agreementLevel` — согласованность источника
- `lambda` — константа сглаживания

### 3. Зачем этот шаг
Источник с тысячами записей заслуживает большего доверия, чем источник с тремя. Но и бесконечный рост уверенности с ростом объёма нежелателен — κ ограничен сверху числом 1. Также согласованность внутри группы (особенно экспертов) снижает «эффективный» объём: 10 спорящих экспертов хуже 6 единодушных.

### 4. Формулы

```
n_eff = relevantCount × agreementLevel
κ     = n_eff / (n_eff + λ)
```

**Почему такая формула?**  
Это сглаженная оценка (Laplace-сглаживание / Bayesian shrinkage). При `n_eff = 0` → `κ = 0` (нет данных — нет уверенности). При `n_eff → ∞` → `κ → 1` (бесконечно много данных — максимальная уверенность). При `n_eff = λ` → `κ = 0.5` (половинная уверенность).

**Смысл λ:** это «полудоверие» — объём данных, при котором уверенность равна 50%.  

**Хардкодированные λ:**

| Источник | λ | Смысл |
|----------|---|-------|
| Вакансии | 15 | 15 вакансий → κ=0.5; стандартный рынок труда |
| Эксперты | 5 | 5 экспертов → κ=0.5; типичная экспертная группа |
| Прогнозы | 2 | 2 прогноза → κ=0.5; прогнозов всегда мало |

**agreementLevel для прогнозов смешанного типа:**  
`n_eff = growthCount + declineCount` (agreementLevel = 1.0, т.к. направление уже разделено).

### 5. Что получаем
`κ ∈ (0, 1)` — скаляр, отражающий «информационную насыщенность» источника.

### 6. Что это даёт
Без этого шага источник с 3 вакансиями и источник с 3000 вакансиями дали бы одинаковый вклад. κ делает оценку пропорциональной объёму данных.

### 7. Ограничения и хардкод
- `λ = 15 / 5 / 2` — **жёстко зашиты** в конструкторах `DemoApplication`. В реальной системе должны вычисляться по медиане объёмов данных.
- Метод расчёта `agreementLevel` для экспертов не реализован в коде — передаётся готовым числом.

### 8. Планы на будущее
- Автоматический подбор `λ` по медиане объёмов (раз в год / при обновлении данных).
- Три метода калибровки `λ`: по медиане, по процентилю, по точности предсказаний на исторических данных.
- Автоматическое вычисление `agreementLevel` для экспертов через дисперсию оценок: `agreement = 1 − (Var / 0.25)`.

## Шаг 2 — Базовое распределение масс (BPA)

### 1. Название
Перевод данных источника в трёхчастное распределение уверенности по теории Демпстера-Шафера.

### 2. Входные данные
- `κ` (из шага 1)
- `averageScore` — средняя оценка важности кластера
- `relevantCount / totalCount` — доля релевантных данных
- Для прогнозов: `avgGrowthScore`, `avgDeclineScore`, `growthCount`, `declineCount`

### 3. Зачем этот шаг
Источники дают разные числа (число вакансий, оценки экспертов, прогнозы роста), и нельзя просто их усредять. BPA переводит все данные в **единую шкалу уверенности** {T, U, F}, которую можно комбинировать по правилу Демпстера.

### 4. Формулы

**Для вакансий и экспертов:**
```
m(T) = κ × averageScore × (relevantCount / totalCount)
m(F) = 0                              (источник не доказывает неважность)
m(U) = 1 − m(T)
```

**Для прогнозов смешанного типа:**
```
m(T) = κ × avgGrowthScore  × (growthCount  / totalCount)   ← сигнал роста
m(F) = κ × avgDeclineScore × (declineCount / totalCount)   ← сигнал спада
m(U) = 1 − m(T) − m(F)
```

**Почему три множителя в `m(T)`?**

| Множитель | Смысл | Что будет без него |
|-----------|-------|-------------------|
| `κ` | Насколько данных достаточно | Мало данных = та же уверенность, что и много |
| `averageScore` | Насколько источник считает кластер важным | Слабый сигнал даст ту же массу, что сильный |
| `relevant/total` | Какая доля данных вообще относится к кластеру | Кластер с 10% упоминаний = тому с 90% |

**Почему `m(F) = 0` для вакансий и экспертов?**  
Эти источники говорят «кластер упомянут» или «эксперт оценил важность». Они не утверждают «кластер точно не нужен». Отсутствие упоминания ≠ доказательство неважности.

### 5. Что получаем
Тройка `(m(T), m(U), m(F))` для каждого источника, где `m(T)+m(U)+m(F) = 1`.

### 6. Что это даёт
Позволяет честно выразить **неопределённость**: источник с малым объёмом данных или слабым сигналом получает высокое `m(U)` — «я не уверен».

### 7. Ограничения и хардкод
- `averageScore` для вакансий **не вычисляется** — передаётся как предположение (0.75–0.85).
- `m(F) = 0` для вакансий и экспертов — **упрощение**. В реальности могут быть явно отрицательные сигналы (эксперты говорят «технология устарела»).
- Нет обработки источников, которые явно говорят «не нужно» — например, отзывы выпускников «этот навык я не применяю».

### 8. Планы на будущее
- Вычислять `averageScore` для вакансий через TF-IDF: частота упоминания навыка, его позиция в тексте (заголовок vs. «желательно»), его связь с уровнем зарплаты.
- Добавить «отрицательные» источники: экспертные оценки устаревших технологий, HR-аналитика.
- Унифицировать BPA для РПД: из объёма часов → `m_std(T)`.

## Шаг 3 — Дисконтирование

### 1. Название
Снижение уверенности каждого источника пропорционально его ненадёжности.

### 2. Входные данные
- `m(T)`, `m(U)`, `m(F)` (из шага 2)
- `weight` — вес надёжности источника `w`

### 3. Зачем этот шаг
Источники не одинаково надёжны: эксперты более точны, чем прогнозы. Без дисконтирования ошибочный источник имел бы такой же вес, как надёжный.

### 4. Формулы

```
m'(T) = w × m(T)
m'(F) = w × m(F)
m'(U) = 1 − m'(T) − m'(F)
```

Эквивалентная форма для `m'(U)`:
```
m'(U) = 1 − w × (1 − m(U))  =  (1−w) + w × m(U)
```

**Почему именно так?**  
Дисконтирование «перекладывает» часть уверенности в неопределённость: `(1−w)×m(T)` уходит из «уверен» в «не знаю». Мы не создаём уверенность в противоположном исходе — источник просто менее надёжен, а не заведомо ошибочен.

**Хардкодированные веса:**

| Источник | w | Обоснование |
|----------|---|-------------|
| Эксперты | 0.90 | Взвешенные профессиональные оценки, минимальные потери |
| Вакансии | 0.80 | Фактические данные, но есть дубли и сезонность |
| Прогнозы | 0.60 | Прогнозы часто ошибаются, рынок IT меняется быстро |

**Что происходит с m(T) после дисконтирования:**

| Источник | m(T) до | w | m'(T) после | Потеря |
|----------|---------|---|-------------|--------|
| Эксперты | 0.520 | 0.9 | 0.468 | −5.2% |
| Вакансии | 0.707 | 0.8 | 0.566 | −14.1% |
| Прогнозы | 0.453 | 0.6 | 0.272 | −18.1% |

### 5. Что получаем
Дисконтированные тройки `(m'(T), m'(U), m'(F))` для каждого источника.

### 6. Что это даёт
Относительное «взвешивание» источников без явного применения формулы взвешенного среднего. Более надёжный источник проигрывает меньше уверенности и в итоге вносит больший вклад при комбинировании.

### 7. Ограничения и хардкод
- `w = 0.80 / 0.90 / 0.60` — **жёстко зашиты**. В статье предполагается их калибровка на исторических данных.
- Веса одинаковы для всех кластеров — в реальности надёжность источника может зависеть от типа технологии (прогнозы точнее для зрелых технологий, чем для новых).

### 8. Планы на будущее
- Калибровка `w` на исторических данных: берём данные прошлых лет, делаем предсказания, сравниваем с реальностью, выбираем `w` с максимальной точностью.
- Адаптивные веса по типу источника и категории кластера.
- Вес на основе метрик качества данных: дублей в вакансиях, ошибки прогнозов за последние N лет.

## Шаг 4 — Комбинирование (правило Демпстера)

### 1. Название
Последовательное слияние мнений трёх источников в одно итоговое распределение.

### 2. Входные данные
Дисконтированные BPA трёх источников: `(m'_vac, m'_exp, m'_fc)`.  
Комбинирование выполняется в два этапа: сначала вакансии ⊕ эксперты, затем результат ⊕ прогнозы.

### 3. Зачем этот шаг
Объединить три независимых мнения в одно, усиливая уверенность там, где источники согласны, и выявляя конфликт там, где они противоречат.

### 4. Формулы

**Коэффициент конфликта K:**
```
K = m₁(T) × m₂(F)  +  m₁(F) × m₂(T)
```
K — доля «несовместимых» пар: один источник говорит «важно», другой «не важно».

**Правило Демпстера (при K < τK = 0.4):**
```
m_out(T) = [m₁(T)×m₂(T) + m₁(T)×m₂(U) + m₁(U)×m₂(T)] / (1 − K)
m_out(U) = [m₁(U)×m₂(U)] / (1 − K)
m_out(F) = [m₁(F)×m₂(F) + m₁(F)×m₂(U) + m₁(U)×m₂(F)] / (1 − K)
```

**Смысл слагаемых в `m_out(T)`:**

| Слагаемое | Ситуация |
|-----------|----------|
| `m₁(T)×m₂(T)` | Оба уверены «важно» → уверенность растёт |
| `m₁(T)×m₂(U)` | Первый уверен, второй не знает → уверенность сохраняется |
| `m₁(U)×m₂(T)` | Первый не знает, второй уверен → уверенность сохраняется |

**Нормировка `1/(1−K)`:**  
Убирает «конфликтную массу» из пространства рассмотрения. При K=0 — нормировка не меняет результат. При K=0.3 — масса увеличивается в ×1.43.

**Интерпретация K:**

| K | Ситуация | Действие |
|---|----------|---------|
| 0 | Источники не противоречат | Демпстер применяется нормально |
| 0.0–0.2 | Незначительный конфликт | ОК |
| 0.2–0.4 | Умеренный конфликт | ОК, но следим |
| ≥ 0.4 | Высокий конфликт | ⚠️ В коде: предупреждение, но **правило Демпстера применяется всё равно** |

### 5. Что получаем
Финальное BPA: `m*(T)`, `m*(U)`, `m*(F)` — единое мнение всех трёх источников.  
Итоговый конфликт: `K = max(K1, K2)`.

### 6. Что это даёт
- При согласии источников: `m*(T)` значительно растёт, `m*(U)` падает → уверенное решение.
- При конфликте: `K` сигнализирует о противоречии → решение откладывается (ВАРИАТИВНОСТЬ).

### 7. Ограничения и хардкод
- При `K ≥ τK = 0.4` **правило Ягера не применяется** — это известный пробел между статьёй и реализацией. Код выводит предупреждение, но продолжает вычислять по Демпстеру.
- Порядок комбинирования фиксирован: VAC ⊕ EXP → (VAC⊕EXP) ⊕ FC. Правило Демпстера коммутативно и ассоциативно, поэтому порядок не влияет на результат — но это явно не документировано.
- `τK = 0.4` — хардкод.

### 8. Планы на будущее
- Реализовать правило Ягера при `K ≥ τK`: конфликтная масса уходит в `m(Θ)` вместо нормировки (см. `ALGORITHM_IDEAL.md`, раздел «Правило Ягера»).
- Рассмотреть альтернативы: правило Дюбуа-Прада (мягче Ягера), TBM (Transferable Belief Model).
- Добавить четвёртый источник STD (РПД) в цепочку комбинирования.

## Шаг 5 — BetP и показатель дефицита Δ

### 1. Название
Преобразование распределения масс в единое число для сравнения с покрытием в учебной программе.

### 2. Входные данные
- `m*(T)`, `m*(U)`, `m*(F)` (из шага 4)
- `supply = currentRpdCoverage` — текущая доля кластера в РПД
- `N = N_CLUSTERS = 25` — число кластеров

### 3. Зачем этот шаг
После шага 4 у нас три числа, а для сравнения с `supply` нужно одно. Нельзя просто взять `m*(T)` — в нём не учтена неопределённость `m*(U)`.

### 4. Формулы

**Pignistic Probability (BetP):**
```
BetP = m*(T)  +  m*(U) / N
```

**Показатель дефицита:**
```
Δ = BetP − supply
```

**Почему BetP, а не просто m*(T)?**  
`m*(U)` — это «нераспределённая» масса. Если мы не знаем, к чему отнести неопределённость, логично предположить, что она равномерно распределяется между всеми N кластерами. Каждый получает добавку `m*(U)/N`.

**Почему N = 25?**  
Принято как оценочное число кластеров в типичной CS-программе. Чем больше N, тем меньше вклад неопределённости в BetP.

### 5. Что получаем
- `BetP ∈ (0, 1)` — вероятностная мера важности кластера.
- `Δ ∈ (−1, 1)` — дефицит: насколько важность превышает текущее покрытие.

**Интерпретация Δ:**

| Δ | Интерпретация |
|---|---------------|
| > +0.15 | Дефицит: рынок требует больше, чем есть в РПД |
| [−0.15; +0.15] | Баланс: программа соответствует требованиям |
| < −0.15 | Избыток: в РПД больше, чем требует рынок |

### 6. Что это даёт
Единственное число, которое можно сравнить с `supply` и автоматически принять решение.

### 7. Ограничения и хардкод
- `N_CLUSTERS = 25` — **жёстко зашит** в `DSTAggregationService`. В реальной системе должен браться из БД (количество кластеров в программе).
- `supply` задаётся вручную — не вычисляется из фактических РПД.
- `τΔ = 0.15` — хардкод.

### 8. Планы на будущее
- `N_CLUSTERS` = реальное число кластеров из БД.
- `supply` = вычислять автоматически из таблиц РПД (часы по кластеру / общий объём программы).
- Рассмотреть альтернативные преобразования: `Pl(T) = 1 − m*(F)` как верхняя оценка, `Bel(T) = m*(T)` как нижняя.

## Шаг 6 — Принятие решения

### 1. Название
Перевод числовых показателей в управленческую рекомендацию.

### 2. Входные данные
- `Δ` — дефицит (из шага 5)
- `K` — конфликт (из шага 4)
- `m*(U)` — итоговая неопределённость (из шага 4)

### 3. Зачем этот шаг
Цель алгоритма — не просто вычислить числа, а дать действенную рекомендацию: усилить кластер, оставить или запросить экспертизу.

### 4. Логика решения

**Три условия:**
```
Условие 1: Δ > τΔ (0.15)     → есть реальный дефицит?
Условие 2: K ≤ τK (0.40)     → источники не противоречат?
Условие 3: m*(U) ≤ τΘ (0.15) → неопределённость приемлема?
```

**Таблица решений:**

| Δ > τΔ | K ≤ τK | m*(U) ≤ τΘ | Решение | Код |
|--------|--------|-------------|---------|-----|
| ✅ | ✅ | ✅ | **УСИЛИТЬ** (жёсткое управляющее воздействие) | `strong` / `moderate` |
| ✅ | ✅ | ❌ | **ВАРИАТИВНОСТЬ** (высокая неопределённость) | `expertise` |
| ✅ | ❌ | любое | **ВАРИАТИВНОСТЬ** (конфликт источников) | `expertise` |
| ❌ (Δ < −τΔ) | — | — | **СОКРАТИТЬ** (избыток в РПД) | `reduce` |
| ❌ | — | — | **СТАБИЛИЗАЦИЯ** (дефицита нет) | `preserve` |

**Градация УСИЛИТЬ:**
- `Δ > 0.5` → «усилить значительно» (+50–100% часов)
- `0.15 < Δ ≤ 0.5` → «усилить умеренно» (+20–50% часов)

**Почему ВАРИАТИВНОСТЬ, а не УСИЛИТЬ при высоком m*(U)?**  
Если данные недостаточно убедительны (`m*(U) > 0.15`), прямое изменение программы рискованно. Алгоритм рекомендует сначала собрать больше данных или провести экспертный анализ.

### 5. Что получаем
- `recommendation` — текстовая строка с рекомендацией.
- `level` — машиночитаемый код (`strong`, `moderate`, `expertise`, `reduce`, `preserve`, `exclude`).

### 6. Что это даёт
Автоматизированный вывод, понятный методистам и руководству кафедры. Ранжирует кластеры по срочности вмешательства.

### 7. Ограничения и хардкод
- `τΔ = 0.15`, `τK = 0.40`, `τΘ = 0.15` — **жёстко зашиты**, не откалиброваны.
- Рекомендация по объёму часов (+20-50%, +50-100%) — **условная**, не учитывает реальный объём РПД.
- Нет перехода на Уровень 2 (анализ отдельных навыков внутри кластера) — `canProceedToLevel2()` есть в коде, но не вызывается.
- Отсутствует история рекомендаций — каждый запуск независим.

### 8. Планы на будущее
- Калибровка порогов `τΔ`, `τK`, `τΘ` на исторических данных через эксперименты с ретроспективными предсказаниями.
- Автоматический запуск Уровня 2 для кластеров с решением УСИЛИТЬ.
- Сохранение рекомендаций в БД с привязкой к дате и версии данных.
- Пересмотр рекомендации при поступлении новых данных (инкрементальное обновление).
